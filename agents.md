# Agents Guide

Этот файл описывает правила работы с проектом для Codex и других AI-агентов. Его нужно читать перед внесением изменений в код.

## 1. Project Context

Проект - локальный видеосайт на FastAPI.

Текущие технические решения:

- backend и frontend находятся в одном FastAPI-приложении;
- HTML отдается через Jinja2 templates;
- CSS и JavaScript лежат в `app/static/`;
- база данных - SQLite;
- ORM - SQLAlchemy;
- авторизация - cookie-сессии через `SessionMiddleware`;
- пароли хешируются через `passlib/bcrypt`;
- видео лежат в локальной папке `video_storage/`;
- большие видео отдаются через streaming с поддержкой HTTP `Range`;
- доступ к видео индивидуальный через таблицу `VideoAccess`.

Ключевые модули:

- `app/main.py` - создание приложения, middleware, templates, routers, startup;
- `app/config.py` - пути, настройки, supported extensions, chunk size;
- `app/database.py` - engine, session factory, dependency `get_db`;
- `app/models.py` - модели `User`, `Video`, `VideoAccess`;
- `app/auth.py` - login/logout, password hashing, current user, admin checks;
- `app/videos.py` - главная страница, просмотр видео, доступы, scan, streaming;
- `app/admin.py` - админ-панель пользователей, видео и прав доступа;
- `app/templates/` - Jinja2 templates;
- `app/static/` - CSS и JS.

## 2. Engineering Principles

- Предпочитать простые решения.
- Не использовать асинхронность без необходимости.
- Не добавлять новые зависимости без явной пользы.
- Следовать существующим паттернам проекта.
- Избегать преждевременной оптимизации.

Практическая трактовка:

- Использовать обычные синхронные FastAPI route handlers, если нет реальной причины для `async`.
- Не добавлять frontend-фреймворки без отдельного решения.
- Не выносить код в новую абстракцию, если это не уменьшает реальное дублирование.
- Не добавлять фоновые очереди, кеши, workers или внешние сервисы для MVP без необходимости.
- Для простых HTML-страниц использовать существующий подход с Jinja2.

## 3. Change Policy

Перед внесением изменений:

1. Изучить существующий код.
2. Найти похожие реализации.
3. Предложить план.
4. Только затем писать код.

После изменений:

1. Проверить типы.
2. Запустить тесты.
3. Проверить импорт модулей.

Если в проекте еще нет настроенных инструментов проверки типов, линтера или тестов, нужно явно указать это в итоговом ответе и выполнить доступные проверки, например:

```powershell
python -m compileall app
python -c "from app.main import app; print(app.title)"
```

## 4. Restrictions

Запрещено:

- менять структуру БД без согласования;
- удалять существующие маршруты;
- хранить пароли в открытом виде;
- отключать проверки доступа;
- загружать файл целиком в память при streaming.

Дополнительные ограничения для этого проекта:

- Не отдавать видео напрямую через `StaticFiles`.
- Не строить путь к видео из пользовательского ввода без проверки по базе.
- Не показывать недоступные видео обычному пользователю.
- Не обходить `require_user`, `require_admin` и `user_can_access_video`.
- Не дублировать логику доступа в шаблонах вместо серверной проверки.
- Не добавлять публичные видео без отдельного изменения product spec.
- Не коммитить `app.db`, `.venv/`, logs или файлы из `video_storage/`.

## 5. Definition of Done

Задача считается завершенной, если:

- код запускается;
- нет ошибок линтера;
- добавлены проверки доступа;
- нет дублирования логики;
- функциональность покрыта тестами.

Для изменений в этом проекте также проверить:

- импорт `app.main` проходит без ошибок;
- route handlers не ломают существующие URL;
- обычный пользователь не получает доступ к чужому видео;
- администратор сохраняет доступ к админ-панели;
- streaming читает файл кусками;
- `Range`-запросы продолжают работать;
- шаблоны открываются без отсутствующих переменных.

## 6. Security Rules

- Пароли хранить только в `password_hash`.
- Для проверки паролей использовать `verify_password`.
- Для создания паролей использовать `hash_password`.
- Админские страницы защищать через `require_admin`.
- Пользовательские страницы защищать через текущую сессию или `require_user`.
- Stream endpoint всегда должен проверять и авторизацию, и доступ к конкретному видео.
- Ошибки доступа не должны раскрывать лишнюю информацию о чужих видео.

## 7. Database Rules

Текущие модели:

- `User`;
- `Video`;
- `VideoAccess`.

Изменения схемы БД требуют согласования до написания кода.

Если изменение все же согласовано:

- описать причину;
- описать миграцию или способ обновления локальной SQLite базы;
- не удалять пользовательские данные без отдельного подтверждения;
- обновить `product_spec.md`, если меняется продуктовая логика.

## 8. Streaming Rules

Видео могут быть размером до 10 ГБ. Поэтому:

- не использовать `file.read()` без ограничения размера;
- не возвращать весь файл как bytes;
- использовать генератор чтения чанками;
- сохранять поддержку `Range`;
- возвращать корректные HTTP-заголовки;
- проверять доступ до открытия файла.

Критичные функции и маршруты:

- `parse_range_header`;
- `iter_file_range`;
- `stream_video`;
- `user_can_access_video`.

Любые изменения в них требуют тестов или ручной проверки:

- обычный запрос без `Range`;
- запрос с `Range: bytes=0-`;
- запрос с ограниченным диапазоном;
- некорректный `Range`;
- доступ к чужому видео.

## 9. UI Rules

- Сохранять простой YouTube-подобный интерфейс.
- Использовать Jinja2 templates.
- Не добавлять React/Vue/Svelte без отдельного решения.
- Не делать маркетинговую landing page вместо основного интерфейса.
- Главная страница должна сразу показывать список видео после входа.
- Страница просмотра должна использовать стандартный HTML5 `<video controls>`.
- Админ-панель должна быть простой и функциональной.

## 10. Testing Guidance

При добавлении тестов покрывать минимум:

- login с правильным паролем;
- login с неправильным паролем;
- запрет админки для обычного пользователя;
- список видео только с доступными видео;
- запрет страницы `/watch/{video_id}` для чужого видео;
- запрет `/videos/{video_id}/stream` для чужого видео;
- корректный `206 Partial Content` для `Range`;
- корректный `416` для неправильного `Range`;
- сортировку по названию;
- сортировку по дате;
- фильтрацию по категории.

Если тестовая инфраструктура отсутствует, сначала предложить минимальный план ее добавления. Не добавлять тяжелые зависимости без явной пользы.
