# Product Spec: локальный видеосайт

## 1. Назначение

Проект представляет собой локальный видеосайт, похожий по базовой логике на YouTube. Сайт запускается на компьютере пользователя, хранит видеофайлы в локальной папке и показывает каждому пользователю только те видео, к которым администратор выдал доступ.

Первый MVP не включает загрузку видео через сайт. Видео добавляются вручную в папку `video_storage/`, после чего администратор запускает сканирование.

## 2. Цели MVP

- Пользователь входит на сайт по логину и паролю.
- После входа пользователь видит только назначенные ему видео.
- Администратор создает пользователей вручную.
- Администратор сканирует папку с видео.
- Администратор редактирует метаданные видео.
- Администратор назначает каждому пользователю индивидуальный список доступных видео.
- Пользователь смотрит видео через стандартный HTML5-плеер.
- Видео до 10 ГБ воспроизводятся без загрузки файла целиком в память.
- Поддерживаются категории и сортировка по названию и дате.

## 3. Не входит в MVP

- Регистрация пользователей.
- Загрузка видео через сайт.
- Комментарии.
- Лайки.
- Подписки.
- Рекомендательная система.
- История просмотров.
- Автоматическая генерация превью.
- Автоматическое определение длительности видео.
- Группы доступа и роли, кроме флага администратора.
- Публичные видео без авторизации.

## 4. Пользователи и роли

### Обычный пользователь

Может:

- войти на сайт;
- выйти из аккаунта;
- видеть список доступных ему видео;
- фильтровать видео по категории;
- сортировать видео по названию и дате;
- открыть страницу просмотра доступного видео;
- смотреть видео через браузерный плеер.

Не может:

- видеть чужие видео;
- открывать чужое видео по прямой ссылке;
- получать поток чужого видео через stream-маршрут;
- заходить в админ-панель.

### Администратор

Может:

- выполнять все действия обычного пользователя;
- открывать админ-панель;
- создавать пользователей;
- создавать пользователей с флагом администратора;
- запускать сканирование папки `video_storage/`;
- видеть все видео в базе;
- редактировать название, описание, автора и категорию видео;
- назначать пользователям доступ к видео.

## 5. Авторизация

Все страницы, кроме `/login`, доступны только после авторизации.

Основные маршруты:

```http
GET  /login
POST /login
POST /logout
GET  /
GET  /watch/{video_id}
GET  /videos/{video_id}/stream
```

Требования:

- пароль проверяется только по хешу;
- пароль нельзя хранить в открытом виде;
- после успешного входа создается cookie-сессия;
- неавторизованный пользователь перенаправляется на `/login`;
- stream-маршрут должен возвращать ошибку авторизации без HTML-редиректа, если это нужно для корректного поведения видеоплеера.

## 6. Главная страница

Маршрут:

```http
GET /
```

Главная страница должна содержать:

- верхнюю панель;
- название сайта;
- имя текущего пользователя;
- кнопку выхода;
- ссылку в админ-панель для администратора;
- список категорий;
- список или сетку доступных видео;
- сортировку по названию;
- сортировку по дате.

Карточка видео должна показывать:

- название;
- автора;
- категорию;
- дату добавления;
- размер файла;
- признак недоступности файла, если файл пропал из папки хранения и это отображается в админке.

Обычный пользователь видит только видео, назначенные ему через `VideoAccess`.

## 7. Страница просмотра

Маршрут:

```http
GET /watch/{video_id}
```

Страница просмотра должна содержать:

- кнопку назад;
- кнопку вперед или переход к следующему доступному видео;
- HTML5-видеоплеер;
- название видео;
- описание;
- автора;
- категорию;
- дату добавления.

Видеоплеер использует стандартные элементы управления браузера:

```html
<video controls>
  <source src="/videos/{video_id}/stream" type="video/mp4">
</video>
```

Должны работать:

- запуск;
- пауза;
- перемотка;
- управление громкостью;
- полноэкранный режим.

## 8. Админ-панель

Админ-панель доступна только пользователям с `is_admin = true`.

Основные маршруты:

```http
GET  /admin
GET  /admin/users
POST /admin/users/create
GET  /admin/videos
POST /admin/videos/scan
POST /admin/videos/{video_id}/update
GET  /admin/access
POST /admin/access/update
```

### Управление пользователями

Администратор должен иметь возможность:

- видеть список пользователей;
- создать пользователя;
- задать логин;
- задать пароль;
- указать, является ли пользователь администратором.

### Управление видео

Администратор должен иметь возможность:

- видеть список видео из базы;
- запустить сканирование папки `video_storage/`;
- видеть имя файла;
- видеть размер файла;
- видеть доступность файла;
- редактировать название;
- редактировать описание;
- редактировать автора;
- редактировать категорию.

### Управление доступом

Администратор должен иметь возможность:

- выбрать пользователя;
- увидеть список всех видео;
- отметить видео, доступные этому пользователю;
- сохранить индивидуальный список доступа.

Доступ не групповой. У каждого пользователя свой список видео.

## 9. Хранение видео

Видео физически хранятся в папке:

```text
video_storage/
```

Поддерживаемые форматы MVP:

- `.mp4`;
- `.webm`;
- `.mkv`, если браузер может воспроизвести файл.

Сканирование должно:

- создать папку `video_storage/`, если ее нет;
- найти поддерживаемые видеофайлы;
- добавить новые видео в базу;
- обновить размер существующих видео;
- пометить отсутствующие файлы как недоступные;
- не удалять записи из базы автоматически.

## 10. База данных

MVP использует SQLite.

Основные модели:

### User

- `id`;
- `username`;
- `password_hash`;
- `is_admin`;
- `created_at`.

### Video

- `id`;
- `filename`;
- `title`;
- `description`;
- `author`;
- `category`;
- `file_path`;
- `size_bytes`;
- `created_at`;
- `updated_at`;
- `is_available`.

### VideoAccess

- `id`;
- `user_id`;
- `video_id`.

Изменения структуры БД требуют отдельного согласования.

## 11. Streaming больших файлов

Маршрут:

```http
GET /videos/{video_id}/stream
```

Требования:

- проверять авторизацию;
- проверять доступ пользователя к видео;
- проверять существование файла;
- поддерживать HTTP-заголовок `Range`;
- отдавать частичный контент со статусом `206 Partial Content`;
- возвращать `Accept-Ranges`;
- возвращать `Content-Range` для частичных ответов;
- возвращать корректный `Content-Length`;
- определять `Content-Type` по файлу;
- читать файл кусками;
- не загружать файл целиком в память.

Ошибки:

- `401 Unauthorized` - пользователь не вошел;
- `403 Forbidden` - нет доступа к видео;
- `404 Not Found` - видео или файл не найден;
- `416 Range Not Satisfiable` - некорректный диапазон байтов.

## 12. Категории и сортировка

Главная страница поддерживает параметры:

```http
/?category=films
/?sort=title&order=asc
/?sort=date&order=desc
/?category=education&sort=title&order=asc
```

Фильтрация:

- по категории;
- только среди доступных текущему пользователю видео.

Сортировка:

- по названию;
- по дате добавления.

Порядок:

- `asc`;
- `desc`.

## 13. Критерии готовности MVP

MVP считается готовым, если:

- сайт запускается локально;
- пользователь может войти по логину и паролю;
- пользователь может выйти из аккаунта;
- администратор может создать пользователя;
- администратор может просканировать папку с видео;
- администратор может редактировать метаданные видео;
- администратор может назначить пользователю доступ к конкретным видео;
- обычный пользователь видит только свои видео;
- обычный пользователь не может открыть чужое видео напрямую;
- обычный пользователь не может получить stream чужого видео;
- видео воспроизводится через браузерный плеер;
- работает перемотка;
- большие файлы отдаются через `Range Requests`;
- сортировка по названию работает;
- сортировка по дате работает;
- фильтрация по категории работает.

## 14. Будущие улучшения

После MVP можно добавить:

- регистрацию пользователей;
- загрузку видео через админку;
- thumbnail-превью;
- определение длительности через `ffmpeg`;
- PostgreSQL вместо SQLite;
- группы доступа;
- поиск по названию и описанию;
- историю просмотров;
- адаптивный дизайн для телефонов;
- отдельный JSON API для будущего frontend-приложения.

## 15. Текущие уточнения продукта

- Главная страница `GET /` доступна без авторизации.
- Неавторизованный пользователь видит заглавную страницу с кнопками входа и регистрации, но не получает доступ к приватным видео.
- Регистрация пользователей доступна через `GET /register` и `POST /register`.
- Для регистрации требуются логин и пароль.
- Новый пользователь создается как обычный пользователь без прав администратора.
- После успешной регистрации создается пользовательская сессия.

## 16. Уточнение текущей главной страницы

- Стартовый экран `GET /` показывает список доступных видеофайлов даже без входа.
- Для гостя карточки видео не открывают просмотр напрямую, а ведут к форме входа.
- Иконка входа находится справа в верхней панели.
- Форма входа может открываться центральной панелью поверх главной страницы.
- Регистрация остается отдельной страницей `/register`.
- Обычный пользователь, зарегистрированный через публичную регистрацию, получает доступ к текущим доступным видео через записи `VideoAccess`.
- Просмотр `/watch/{video_id}` и поток `/videos/{video_id}/stream` остаются защищенными авторизацией и проверкой доступа.

## 17. Будущая функция: доступ с других устройств

- Добавить поддерживаемый режим запуска сервера для подключения с других устройств в локальной сети.
- Для локальной сети сервер должен уметь запускаться с привязкой к `0.0.0.0`.
- Документировать URL вида `http://<local-ip>:8000/` для подключения с другого устройства.
- Документировать требования к Windows Firewall и настройкам сети.
- Доступ из интернета или другой внешней сети не включать без отдельной проработки безопасности, HTTPS, секретов сессии и сетевой схемы.

## 18. Загрузка и удаление пользовательских видео

- Авторизованный пользователь видит кнопку `Загрузить` справа в верхней панели, левее имени пользователя.
- По нажатию на `Загрузить` открывается центральная панель загрузки.
- Панель загрузки поддерживает выбор файла и drag and drop.
- Пользователь может указать название видео при загрузке.
- Сервер принимает загрузку через `POST /videos/upload`.
- Сервер сохраняет видео в `video_storage/` чанками, не читая весь файл в память.
- Для файлов, добавленных вручную в `video_storage/`, поддерживаются только расширения из `SUPPORTED_VIDEO_EXTENSIONS`.
- Для пользовательской загрузки поддерживаются входные расширения из `UPLOAD_VIDEO_EXTENSIONS`, а итоговый файл сохраняется как `.mp4`.
- Текущее ограничение: видео, закодированные в H.265/HEVC, на сайте не воспроизводятся надежно.
- Поддержку H.265/HEVC нужно устранить позже через серверное перекодирование в браузерный формат, например `.mp4` с H.264 video, AAC audio, `yuv420p` и `faststart`.
- Для будущего перекодирования потребуется доступный `ffmpeg` в `PATH` или через переменную окружения `LOCALTUBE_FFMPEG_PATH`.
- Для видео генерируется автоматическое превью из первого кадра через `ffmpeg`.
- Превью хранится как производный JPEG-файл в `video_storage/thumbnails/{video_id}.jpg` и не требует отдельного поля в БД.
- Главная страница показывает превью на карточке видео; если превью еще не создано или `ffmpeg` недоступен, показывается fallback-карточка.
- Доступ к превью идет через `GET /videos/{video_id}/thumbnail`; для авторизованного пользователя применяется та же проверка `VideoAccess`, что и для просмотра видео.
- Пользователь может переключать светлую и темную тему интерфейса; темная тема не должна использовать чистый черный цвет как основной фон.
- Пользователь может скрывать и возвращать боковое меню на главной странице.
- Для каждого аккаунта сохраняется отдельная история просмотров.
- Раздел `История просмотров` доступен в боковом меню главной страницы и может быть очищен пользователем.
- Раздел `Смотреть позже` доступен в боковом меню и наполняется кнопкой `Смотреть позже` под видео.
- Раздел `Понравившиеся видео` показывает видео, которым пользователь поставил лайк.
- Лайк и дизлайк являются взаимоисключающими реакциями: повторное нажатие на активную реакцию убирает ее, выбор другой реакции заменяет старую.
- Кнопка `Смотреть позже` является переключателем: добавляет видео в одноименный список или убирает его оттуда.
- На странице просмотра доступны лайк, дизлайк, комментарии и кнопка `Поделиться`.
- Лайк, дизлайк, `Смотреть позже`, добавление и удаление комментариев на странице просмотра должны отправляться на сервер без полной перезагрузки страницы; обычный POST-redirect остается fallback-поведением без JavaScript.
- Комментарии могут удалять автор комментария, администратор и автор видео.
- В блоке похожих видео на странице плеера отображаются те же thumbnail-превью, что и на главной странице.
- Кнопки удаления видео показываются только на странице просмотра и не показываются на главной странице.
- После загрузки создается запись `Video`, а автором считается текущий пользователь.
- После загрузки создаются записи `VideoAccess` для всех обычных пользователей, чтобы пользователи видели видео друг друга; администратор видит видео без отдельной записи доступа.
- Для уже существующих обычных пользователей недостающие доступы к пользовательским загрузкам синхронизируются при открытии главной страницы.
- Сервер поддерживает загрузку видео до `10 ГБ` и во время сохранения контролирует размер по записанным чанкам.
- После успешной загрузки пользователь перенаправляется на страницу просмотра загруженного видео.
- Страница просмотра не должна жестко указывать `video/mp4` для всех файлов; браузер должен получать фактический MIME от stream endpoint.
- При отправке формы загрузки пользователь видит анимацию загрузки не менее 1 секунды.
- Удалять видео может только администратор или автор видео.
- Удаление выполняется через `POST /videos/{video_id}/delete`.
- При удалении удаляется файл из `video_storage/` и запись из базы.
- Перед удалением со страницы просмотра frontend должен остановить HTML5-плеер, чтобы освободить файл на Windows.
- При удалении сервер прерывает активные stream-ответы для удаляемого `video_id`.
- Если файл все равно занят системой и не может быть удален физически сразу, сервер скрывает видео из интерфейса через `is_available = false`, удаляет доступы, возвращает пользователя на главную без `500 Internal Server Error` и продолжает удалять файл из `video_storage/` в фоновой retry-задаче.
- После успешного фонового удаления файла сервер удаляет запись `Video` из базы.
