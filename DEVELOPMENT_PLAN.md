# План разработки локального видеосайта

## 1. Цель проекта

Создать локальный сайт, похожий по базовой логике на YouTube:

- пользователь входит по логину и паролю;
- после авторизации видит только доступные ему видео;
- видео хранятся в локальной папке на компьютере;
- администратор управляет пользователями, видео и правами доступа;
- пользователь может смотреть видео через браузерный видеоплеер;
- поддерживаются категории и сортировка по названию и дате;
- сайт рассчитан на большие видеофайлы до 10 ГБ.

На первом этапе загрузка видео через сайт не реализуется. Видео добавляются вручную в папку хранения, после чего администратор запускает сканирование.

## 2. Технологический стек

Backend и frontend будут находиться в одном FastAPI-приложении.

Основной стек:

- Python;
- FastAPI;
- Jinja2 templates;
- HTML;
- CSS;
- JavaScript;
- SQLite;
- SQLAlchemy;
- passlib/bcrypt для хеширования паролей;
- cookie-сессии для авторизации.

React, Vue и другие frontend-фреймворки на первом этапе не используются.

## 4. Основные сущности

### User

Пользователь сайта.

Поля:

- `id`;
- `username`;
- `password_hash`;
- `is_admin`;
- `created_at`.

Пароли нельзя хранить в открытом виде. В базе должен храниться только хеш пароля.

### Video

Видео, найденное в локальной папке.

Поля:

- `id`;
- `filename`;
- `title`;
- `description`;
- `author`;
- `category`;
- `file_path`;
- `size_bytes`;
- `created_at`;
- `updated_at`;
- `is_available`.

На первом этапе файл физически лежит в `video_storage/`, а в базе хранится информация о нем.

### VideoAccess

Связь между пользователем и видео.

Поля:

- `id`;
- `user_id`;
- `video_id`.

Доступ работает по принципу: каждому пользователю назначается свой список доступных видео.

## 5. Авторизация

Все страницы сайта, кроме страницы входа, должны быть доступны только после авторизации.

Маршруты:

```http
GET  /login
POST /login
POST /logout
GET  /
GET  /watch/{video_id}
```

Логика:

1. Пользователь открывает сайт.
2. Если он не авторизован, сервер перенаправляет его на `/login`.
3. Пользователь вводит логин и пароль.
4. Сервер проверяет пароль по хешу из базы.
5. После успешного входа создается сессия.
6. На главной странице пользователь видит только доступные ему видео.
7. При попытке открыть чужое видео напрямую сервер возвращает `403 Forbidden`.

Проверка доступа должна выполняться не только на HTML-странице, но и на маршруте потоковой отдачи видео.

## 6. Пользовательский интерфейс

Дизайн должен быть примерно похож на YouTube, но без сложного frontend-фреймворка.

### Главная страница

Главная страница `/` содержит:

- верхнюю панель;
- название сайта;
- имя текущего пользователя;
- кнопку выхода;
- ссылку в админ-панель для администратора;
- боковую панель с категориями;
- список или сетку доступных видео;
- сортировку по названию;
- сортировку по дате добавления.

Карточка видео содержит:

- название;
- автора;
- категорию;
- дату добавления;
- размер файла, если нужно;
- ссылку на страницу просмотра.

### Страница просмотра

Страница `/watch/{video_id}` содержит:

- кнопку назад;
- кнопку вперед;
- видеоплеер;
- название видео;
- описание;
- автора;
- категорию;
- дату добавления.

Видеоплеер должен использовать стандартный HTML5 `video`:

```html
<video controls>
  <source src="/videos/{video_id}/stream" type="video/mp4">
</video>
```

Стандартные элементы управления браузера должны обеспечивать:

- запуск;
- паузу;
- перемотку;
- управление громкостью;
- полноэкранный режим.

## 7. Админ-панель

Админ-панель доступна только пользователям с `is_admin = true`.

Основные маршруты:

```http
GET  /admin
GET  /admin/users
POST /admin/users/create

GET  /admin/videos
POST /admin/videos/scan
POST /admin/videos/{video_id}/update

GET  /admin/access
POST /admin/access/update
```

На первом этапе админ-панель должна уметь:

1. Показывать список пользователей.
2. Создавать пользователей вручную.
3. Показывать список видео из базы.
4. Запускать сканирование папки `video_storage/`.
5. Редактировать метаданные видео:
   - название;
   - описание;
   - автор;
   - категория.
6. Назначать каждому пользователю список доступных видео.

Страницу регистрации пока не делаем. Пользователей создает администратор.

## 8. Хранение и сканирование видео

Видео физически хранятся в папке:

```text
video_storage/
```

Администратор добавляет видеофайлы в эту папку вручную.

После этого администратор нажимает кнопку сканирования в админ-панели.

Сервер должен:

1. Найти поддерживаемые видеофайлы.
2. Проверить, есть ли они уже в базе.
3. Для новых файлов создать записи в таблице `Video`.
4. Для существующих файлов обновить размер и доступность.
5. Не удалять запись из базы автоматически, если файл пропал.

На MVP основной формат:

- `.mp4`.

Дополнительно можно поддержать:

- `.webm`;
- `.mkv`, если браузер сможет воспроизводить файл.

## 9. Потоковая отдача видео

Так как планируются файлы до 10 ГБ, нельзя загружать весь файл в память.

Маршрут:

```http
GET /videos/{video_id}/stream
```

Обязательная логика:

1. Проверить, что пользователь авторизован.
2. Проверить, что у пользователя есть доступ к видео.
3. Проверить, что файл существует.
4. Прочитать HTTP-заголовок `Range`.
5. Отдать нужный диапазон байтов.
6. Вернуть статус `206 Partial Content` для частичного ответа.
7. Вернуть корректные заголовки.

Важные заголовки:

```http
Accept-Ranges: bytes
Content-Range: bytes start-end/total
Content-Length: chunk_size
Content-Type: video/mp4
```

Это необходимо для:

- быстрой перемотки;
- воспроизведения больших файлов;
- частичной загрузки видео браузером;
- работы стандартного HTML5-плеера.

Ошибки:

- `401 Unauthorized` - пользователь не вошел;
- `403 Forbidden` - нет доступа к видео;
- `404 Not Found` - видео или файл не найден;
- `416 Range Not Satisfiable` - некорректный диапазон байтов.

## 10. Категории и сортировка

Главная страница должна поддерживать фильтрацию и сортировку.

Примеры URL:

```http
/?category=films
/?sort=title&order=asc
/?sort=date&order=desc
/?category=education&sort=title&order=asc
```

Фильтрация:

- по категории;
- только среди видео, доступных текущему пользователю.

Сортировка:

- по названию;
- по дате добавления.

Параметр `order`:

- `asc` - по возрастанию;
- `desc` - по убыванию.

## 11. Этапы разработки

### Этап 1. Каркас проекта

- Создать структуру папок.
- Добавить `requirements.txt`.
- Создать базовое FastAPI-приложение.
- Подключить Jinja2 templates.
- Подключить статические файлы.
- Сделать базовый layout `base.html`.

### Этап 2. База данных

- Подключить SQLite.
- Настроить SQLAlchemy.
- Создать модели:
  - `User`;
  - `Video`;
  - `VideoAccess`.
- Создать механизм инициализации базы.

### Этап 3. Авторизация

- Реализовать хеширование паролей.
- Сделать страницу входа.
- Сделать обработку login/logout.
- Сделать получение текущего пользователя из сессии.
- Закрыть все основные страницы от неавторизованных пользователей.
- Создать первого администратора.

### Этап 4. Главная страница

- Сделать страницу списка видео.
- Показывать только видео, доступные текущему пользователю.
- Добавить фильтр по категории.
- Добавить сортировку по названию и дате.
- Добавить базовый YouTube-подобный внешний вид.

### Этап 5. Админ-панель пользователей

- Создать страницу списка пользователей.
- Добавить форму создания пользователя.
- Добавить флаг администратора.
- Запретить доступ в админку обычным пользователям.

### Этап 6. Сканирование видео

- Создать папку `video_storage/`.
- Реализовать поиск видеофайлов.
- Добавить кнопку сканирования в админ-панели.
- Создавать записи в базе для новых видео.
- Обновлять размер и доступность существующих видео.

### Этап 7. Админ-панель видео

- Показывать список видео.
- Добавить редактирование:
  - названия;
  - описания;
  - автора;
  - категории.
- Показывать размер файла и имя файла.

### Этап 8. Управление доступом

- Создать страницу назначения доступа.
- Для каждого пользователя показывать список видео.
- Сохранять выбранные видео в `VideoAccess`.
- Проверять доступ на главной странице.
- Проверять доступ на странице просмотра.
- Проверять доступ на stream-маршруте.

### Этап 9. Страница просмотра

- Создать страницу `/watch/{video_id}`.
- Добавить HTML5-видеоплеер.
- Добавить кнопки назад и вперед.
- Добавить отображение метаданных видео.
- Обработать ошибки доступа и отсутствующего видео.

### Этап 10. Range streaming

- Реализовать потоковую отдачу файла.
- Поддержать заголовок `Range`.
- Читать файл кусками.
- Не загружать файл целиком в память.
- Проверить перемотку видео.
- Проверить работу на большом файле.

### Этап 11. Тестирование

Проверить:

- вход с правильным логином и паролем;
- отказ при неправильном пароле;
- обычный пользователь не видит админку;
- обычный пользователь видит только назначенные видео;
- прямой URL чужого видео возвращает `403`;
- stream чужого видео возвращает `403`;
- перемотка работает;
- видео не загружается целиком в память;
- сортировка по названию работает;
- сортировка по дате работает;
- фильтрация по категории работает.

## 12. Что не входит в первый MVP

На первом этапе не делаем:

- загрузку видео через сайт;
- регистрацию пользователей;
- комментарии;
- лайки;
- подписки;
- рекомендации;
- историю просмотров;
- автоматическую генерацию превью;
- автоматическое определение длительности видео;
- роли и группы доступа;
- публичные видео без авторизации.

Эти функции можно добавить после стабильной версии MVP.

## 13. Будущие улучшения

После MVP можно добавить:

- страницу регистрации;
- загрузку видео через админку;
- обработку видео через фоновые задачи;
- генерацию thumbnail-превью;
- определение длительности через `ffmpeg`;
- PostgreSQL вместо SQLite;
- роли пользователей;
- группы доступа;
- поиск по названию и описанию;
- историю просмотров;
- адаптивный дизайн для телефонов;
- отдельный API для будущего frontend-приложения.

## 14. Критерии готовности MVP

MVP можно считать готовым, если:

- сайт запускается локально;
- пользователь может войти по логину и паролю;
- администратор может создать пользователя;
- администратор может просканировать папку с видео;
- администратор может редактировать метаданные видео;
- администратор может назначить пользователю доступ к конкретным видео;
- обычный пользователь видит только свои видео;
- обычный пользователь может открыть страницу просмотра;
- видео воспроизводится через браузерный плеер;
- работает перемотка видео;
- большие файлы отдаются через `Range Requests`;
- пользователь не может открыть чужое видео напрямую.
